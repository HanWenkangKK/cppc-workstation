做一个页面，包括输入和输出两个部分，可以分居页面两侧（我也不知道）
输入部分：
1、勾选。需要请求后端接口拿到评估树，生成选项
2、语音录入：需要使用科大讯飞的语音api
3、文字输入（可能是患者的基础信息 、异常症状补充说明、治疗师临床经验等）
4、图片输入（可能是患者的基础信息 、异常症状补充说明、治疗师临床经验等）
输出部分：
1、生成文件区：会有多个文件被输出，如评估报告、健康方案等。可以每个文件一个图标的形式展示在这里，可以点击每个文件来查看。
2、文件展示区：点击文件后，展示文件详情的区域。
由于输出内容需要支持编辑，还可能要做特殊的可视化处理，所以后端不能真的返回一个pdf文件，后端给到的数据会按照评估报告的标题、正文、图表等内容对把文件组织成数据结构，前端接收后，不同类型的数据按照不同的方式展示，（如标题需要居中、表格需要画echarts图表）。需要支持各部分内容的修改。修改后支持保存。
我个人的一些想法：评估报告等文件的的展示需要可视化、需要美观，但是数据又要支持修改，如果把展示和修改放在同一个位置可能没法兼顾，所以我的想法是，在展示页面点击想要修改的部分之后，会单独有一个区域把点击部分以可编辑文本的形式展示，用户在这个区域对数据（文本）进行修改后点击确定，即可更新展示区的文本或者图表视图。
AI建议：
1. 对前端同事的补充：交互与协议
你的想法很好，但要落地，前端需要明确以下三个“底层规则”：
数据结构的“组件化”约定： 告诉前端：“后端给你的不是一段 HTML，而是一个 JSON 数组。”
示例：[{type: 'text', content: '...'}, {type: 'chart', data: {...}}, {type: 'table', rows: []}]。
要求：前端需要写一个“渲染器”，根据 type 自动匹配是显示文本组件、ECharts 组件还是表格组件。
“局部编辑”的反馈机制：
既然你提出点击即修改，需要明确：修改后是实时保存还是统一点击底部的“保存”按钮？建议一期先做“统一保存”，降低后端压力。
图片处理逻辑：
输入端的图片，前端需要处理压缩和预览。问他是否需要你提供 MinIO/OSS 的直传接口，还是他传给后端中转。
给前端的 Action Items:
架构：采用 Vue3 + Element Plus，页面做 Scoped CSS 隔离。
核心组件：开发一个 SmartRenderer.vue（智能渲染器），用于解析后端传来的 JSON 结构并转换成文字/图表。
交互设计：实现你提到的“点击-弹窗/侧边栏修改-同步回主视图”的逻辑。
接口对接：集成讯飞 SDK，处理图片上传预览。
2. 对 AI 同事的补充：数据颗粒度与格式
AI 同学最容易把表做得“太散”或“太死”，你需要约束他：
唯一 ID 锚点：
要求：他提炼的映射表，每一行必须有一个唯一的业务 ID。
原因：这样你后端入库后，前端传个 1001，你就能秒回对应的机制和方案，而不需要去猜是哪个文本。
输出内容的“标签化”：
要求：在方案列中，除了文字，最好能标注出关键词。
示例：方案列不仅仅是“电刺激”，而是 { action: '电刺激', frequency: '20min', position: '拮抗肌' }。
意义：这能支撑你刚才想让前端实现的“可视化展示”（比如把动作名称加粗显示）。
RAG 的兜底逻辑：
如果用户输入的内容在映射表里找不到怎么办？要求 AI 同学在工作流里写好“模糊匹配”或“语义检索”的逻辑。
给 AI 的 Action Items:
数据清洗：从《手册》中提炼出《CPPC 业务逻辑联动表.xlsx》。
工作流开发：在 Coze 中搭建 RAG 流程，输入为 Tag_IDs，输出为符合后端定义的 JSON 结构。
Prompt 调优：确保生成内容包含“医疗合规性”校验，不准出现夸大词汇。
