## 1. 当前状态快照

- 项目阶段：一期 MVP 启动期
- 当前判断：项目“宪法”已冻结，业务闭环与技术边界已明确，尚处于正式开发启动前的任务拆解阶段
- 当前目标：在 4 周内跑通“评估录入 -> AI 生成 -> 在线修改 -> 确认归档 -> 模拟 PDF 导出”的最小闭环
- 当前原则：优先完成核心闭环，不在一期内扩展复杂权限、多机构租户、历史版本重构

## 2. 已冻结共识

### 2.1 业务范围

一期只做：

- 患者信息管理（极简）
- CPPC 评估树展示与勾选录入
- 对接 Coze 工作流生成评估报告/康复方案
- 结构化内容的存储、读取、局部修改与保存
- 最终确认并模拟 PDF 导出

一期不做：

- 复杂权限系统
- 多机构租户切换
- 历史版本大重构

### 2.2 成功标准

- 治疗师能在 1 分钟内完成勾选并看到 AI 生成的结构化方案
- 方案任意局部内容被修改后，刷新页面仍能保留修改结果

### 2.3 技术与架构约束

- 后端：Spring Boot 3.x
- 数据库：MySQL 8.x
- AI：通过 API 调用 Coze 工作流，不落本地模型
- 后端职责：权限控制、数据持久化、上下文组装、结果校验
- 前后端交互：后端返回结构化组件数据，不直接返回 PDF
- 代码分层：`Controller -> Service -> Mapper/Repository`
- 数据原则：
  - AI 结果不得透传，必须经后端校验并持久化
  - 所有人工修改必须留痕，保留原值与现值

### 2.4 数据契约

- 输入端标准：`Patient_ID + List<Tag_ID> + 补充文本/语音/图片`
- 输出端标准：`Component_List`
- 报告内容存储格式：JSON 数组，支持 `text/chart/table` 等组件类型
- 局部修改机制：必须支持基于 `element_id` 的 `PATCH` 更新

## 5. 按角色拆解的当前任务

### 5.1 后端 / 项目负责人

本角色负责“数据底座 + 契约定义 + 系统中转”。

当前待办：

- 输出数据库初稿：`patient_info`、`assessment_record`、`cppc_tags`、`report_record`、`modify_log`
- 输出接口契约初稿：
  - 获取评估树
  - 创建评估记录
  - 调用 AI 生成报告/方案
  - 获取结构化报告详情
  - 基于 `element_id` 的局部修改保存
  - 确认归档 / 模拟导出
- 明确 AI 接口输入输出 JSON 规范
- 负责 AI 输出的后端校验、持久化和留痕

### 5.2 前端

本角色负责“工作站页面 + 结构化渲染 + 编辑体验”。

当前待办：

- 实现输入区与输出区双栏工作站页面
- 对接评估树接口并生成勾选交互
- 接入语音录入、文本输入、图片输入
- 实现“文件列表 + 文件详情”两段式输出区
- 实现结构化组件渲染器，支持 `text/chart/table`
- 实现“点击展示区 -> 侧边栏/弹窗编辑 -> 回写主视图”的修改流程
- 预留最终确认与模拟 PDF 导出入口

### 5.3 AI

本角色负责“知识准备 + Coze 工作流 + 结构化输出”。

当前待办：

- 依据专业材料提炼“评估-机制-方案”三位一体映射关系
- 建立唯一业务 ID 体系，保证可回写数据库
- 在 Coze 中完成一期工作流搭建
- 约束输出为结构化 JSON，而不是长文本
- 基于案例库进行 Few-shot 调优
- 基于禁用词/敏感词要求做合规性约束

## 7. 当前主要风险

- 风险 1：甲方核心资料不到位，导致评估树和知识库无法按时落地
- 风险 2：AI 输出如果不按结构化 JSON 约束返回，会直接拖慢前后端联调
- 风险 3：前端渲染与编辑逻辑若未提前统一组件协议，后期返工成本高
- 风险 4：若不尽快冻结 `PATCH` 修改模型和留痕格式，刷新保留状态的核心目标无法保障
