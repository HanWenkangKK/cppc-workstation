# PROGRESS

> 更新时间：2026-03-31
>
> 文档用途：记录 CPPC 神经康复 AI 智能工作站一期 MVP 的当前共识、实际开发进度、当前实现边界与下一阶段工作准备。

## 1. 当前状态快照

- 项目阶段：一期 MVP 开发中
- 当前判断：后端“患者评估 -> AI 生成报告”最小闭环已完成，项目已从“启动前设计阶段”进入“前后端联调与真实 AI 接入准备阶段”
- 当前目标：在 4 周内跑通“评估录入 -> AI 生成 -> 在线修改 -> 确认归档 -> 模拟 PDF 导出”的最小闭环
- 当前原则：优先完成核心闭环，不在一期内扩展复杂权限、多机构租户、历史版本重构

## 2. 已冻结共识

### 2.1 业务范围

一期只做：

- 患者信息管理（极简）
- CPPC 评估树展示与勾选录入
- 对接 Coze 工作流生成评估报告/康复方案
- 结构化内容的存储、读取、局部修改与保存
- 最终确认并模拟 PDF 导出

一期不做：

- 复杂权限系统
- 多机构租户切换
- 历史版本大重构

### 2.2 成功标准

- 治疗师能在 1 分钟内完成勾选并看到 AI 生成的结构化方案
- 方案任意局部内容被修改后，刷新页面仍能保留修改结果

### 2.3 技术与架构约束

- 后端：Spring Boot 3.x
- 数据库：MySQL 8.x
- AI：通过 API 调用 Coze 工作流，不落本地模型
- 后端职责：权限控制、数据持久化、上下文组装、结果校验
- 前后端交互：后端返回结构化组件数据，不直接返回 PDF
- 代码分层：`Controller -> Service -> Mapper/Repository`
- 数据原则：
  - AI 结果不得透传，必须经后端校验并持久化
  - 所有人工修改必须留痕，保留原值与现值

### 2.4 数据契约

- 输入端标准：`Patient_ID + List<Tag_ID> + 补充文本/语音/图片`
- 输出端标准：`Component_List`
- 报告内容存储格式：JSON 数组，支持 `text/chart/table` 等组件类型
- 局部修改机制：必须支持基于 `element_id` 的 `PATCH` 更新

### 2.5 当前实现边界

- 当前已完成的最小闭环为：
  - 创建患者
  - 创建评估记录
  - 获取评估树
  - 提交评估标签
  - 生成 AI 报告
- 当前报告生成链路已打通，但 AI 侧仍为后端 mock 实现
- 当前已预留 `RemoteAiService` 抽象层，后续替换为真实 Coze HTTP 调用时，不应破坏前端接口契约
- 当前 `report_record.content` 已作为报告展示的数据底座
- 当前前端至少需要支持的组件类型为：`title`、`text`、`table`
- 当前尚未实现：
  - 报告局部修改 `PATCH`
  - 报告详情回显类接口
  - 真实 Coze 接入
  - PDF 文件落库或最终导出链路

## 6. 按角色拆解的当前任务

### 6.1 后端 / 项目负责人

本角色负责“数据底座 + 契约定义 + 系统中转”。

当前待办：

- 冻结当前 5 个接口是否作为一期联调基线版本
- 决定是否补充：
  - 获取患者详情接口
  - 获取评估详情接口
  - 获取报告详情接口
- 设计并实现报告局部修改 `PATCH` 接口
- 设计修改留痕模型，补齐 `modify_log` 或等价方案
- 明确统一错误码表，而不是仅保留当前示例错误码
- 将 `RemoteAiService` 从 mock 替换为真实 Coze 调用
- 根据 Coze 实际需要，判断是否在后端载荷中补充标签名称、标签路径等语义字段

### 6.2 前端

本角色负责“工作站页面 + 结构化渲染 + 编辑体验”。

当前待办：

- 基于当前 5 个接口完成首轮页面联调
- 完成患者创建与评估创建页面流程
- 对接评估树接口并完成勾选提交
- 实现报告展示区域的结构化组件渲染
- 当前至少支持：`title`、`text`、`table`
- 确认是否需要：
  - 页面回显所需的详情接口
  - 报告生成中的 loading / polling 机制
  - 预留报告局部编辑交互

### 6.3 AI / Coze

本角色负责“真实 AI 生成逻辑 + Coze 工作流 + 结构化输出”。

当前待办：

- 接收后端固定输入结构：`patient + assessment + reportType`
- 保证稳定返回严格 JSON，而不是自然语言长文本
- 当前组件类型先限制在：`title`、`text`、`table`
- 明确真实 Coze HTTP 接口、认证方式、超时时间与失败重试建议
- 确认是否需要后端额外传入标签名称、标签路径，而不仅是 `tagIds`
- 基于模板、案例库、few-shot 优化文案质量与知识库召回效果

## 8. 当前主要风险

- 风险 1：当前 AI 生成为 mock，真实 Coze 接入后若不能稳定返回严格 JSON，将直接阻塞联调
- 风险 2：当前仅完成“生成报告”接口，若不尽快补齐详情回显接口，前端页面状态恢复会受限
- 风险 3：`PATCH` 修改与修改留痕尚未落地，距离“一期成功标准”仍有关键缺口
- 风险 4：若 `cppc_tags` 缺少真实业务数据，评估树只能联调演示，无法进入真实业务测试
- 风险 5：若 Coze 侧只接受 `tagIds` 而不补语义解析，生成质量与稳定性可能不足

## 9. 下一阶段设计准备

本阶段需要优先冻结以下设计决策：

1. 当前 5 个接口是否作为一期联调冻结版本，还是补充详情接口后再冻结。
2. `report_record.content` 是否正式作为后续报告编辑功能的数据底座。
3. 报告局部修改是否在一期内强制落地，并采用何种 `PATCH` 载荷格式。
4. 是否提前制定统一错误码表。
5. Coze 真实接入是否固定为由 `cppc-ai-agent` 直接发起 HTTP 调用。
6. Coze 请求体是否需要从 `tagIds` 升级为“`tagIds + 标签语义`”。
